## 分析
### 1. I -> S (insert)
- **現象**: 在模擬期間出現大量 'state change: I -> S (insert)'
- **Explain**: cache block from Invalid(I) state to Shared(S) state, 通常會發生在cache miss、從L2 cache傳入L1 cache.
- **意義**: 符合MESI 協議，多核心可以讀同一筆資料，但不可以同時寫。這是多核心system中，資料被多個core讀取時的行為。

### 2. S/M/E -> I (invalidate)
- **現象**：state change: S -> I (invalidate)`、`E -> I (invalidate)`、`M -> I (invalidate)
- **Explain**:Cache block invalidation (資料被寫入or cache replacememnt).According to MESI, 當某一個core 寫資料時，其他core的cache必須把這個block標記為invalid，確保不同core的cache資料保持一致。當cache滿時，要放新的資料，舊的block會被替換，狀態也會變為invalid。
- **意義**:這為了確保資料可以一致，防止race condition 的情況發生。

### 3. Block 地址分布
- **位置**:Block 0, 0x1, 0xe, 0xd, 0x7, 0xffffffffffffffff 等
- **Explain**: 0xffffffffffffffff被標示為無效或者是未初始化，不會對應到實際記憶體位置，有助於快取控制器管理和查找。

- **架構目標已達成**：已成功在 gem5 下建立雙核心、私有 L1、共用 L2 架構，並能記錄快取區塊的 MESI 狀態變化。
- **log 證明快取一致性協議正常運作**，I->S 代表資料共享載入，S/M/E->I 代表資料無效化。
